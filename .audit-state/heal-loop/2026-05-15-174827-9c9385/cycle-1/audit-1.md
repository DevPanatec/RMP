# Audit 1 cycle 1 — Backend / Data Layer

## P0
- [cleaning-01] convex/cleaning.ts:180 — `addAssignment` mutation NO llama `requireWriteRole`; enterprise/viewer pueden crear cleaning_assignments. Auth bypass para escritura. Fix: Añadir `await requireWriteRole(ctx);` antes de `requireModulo` en handler.
- [route_progress-01] convex/route_progress.ts:38-39 — `list` query con `scopedOrg=null` (admin/enterprise sin organizacion_id) retorna `all` (TODA la tabla cross-org). Scope leak. Fix: Si `scopedOrg===null && !scope.isSuperAdmin && !scope.isCrossOrgViewer` → `return []`.
- [rutas-01] convex/rutas.ts:73-75 — `list` fall-through devuelve `await ctx.db.query("rutas").collect()` (TODAS las rutas) cuando `scopedProject` y `scopedOrg` son null y user no es super_admin. Scope leak. Fix: Reemplazar return final por `if (scope.isSuperAdmin || scope.isCrossOrgViewer) return all; return [];`.
- [asignaciones-01] convex/asignaciones.ts:19-25 — `list` con `scopedOrg=null` retorna `all` (TODAS las asignaciones cross-org) si user no es super_admin y no tiene org. Fix: tras `getScopedOrgId`, si null y no super_admin/cross-org → `return []`.
- [cleaning-02] convex/cleaning.ts:11-23 — `listSalas` con `scoped=null` retorna `ctx.db.query("salas").withIndex("by_activo", q=>q.eq("activo", true)).collect()` sin filtrar por org. Cualquier user autenticado ve salas de TODAS las orgs. Fix: filtrar por scope.organizacionId cuando no es super_admin/cross-org.
- [fumigaciones-01] convex/fumigaciones.ts:19-23 — `listLugares` mismo bug: `scoped=null` retorna lugares activos de TODAS las orgs. Fix: filtrar por scope.organizacionId post-query.
- [asignaciones-02] convex/asignaciones.ts:304 — `_migrationBackfillOrganizacionId` mutation SIN auth gate ni env gate. Cualquier user autenticado puede ejecutarla. Mutación destructiva (escribe organizacion_id en filas con datos sensibles). Fix: Añadir `await requireSuperAdmin(ctx); assertBackfillAllowed();` al inicio del handler.
- [cleaning-03] convex/cleaning.ts:378 — `_migrationBackfillOrganizacionId` sin auth gate. Misma vulnerabilidad. Fix: añadir `await requireSuperAdmin(ctx)` + env check.
- [empleados-01] convex/empleados.ts:232 — `_migrationBackfillOrganizacionId` sin auth gate. Asigna `defaultOrgId` arbitrariamente si solo hay 1 org en DB — puede ligar empleado a org incorrecta en multi-tenant. Fix: añadir `await requireSuperAdmin(ctx)` + env check.
- [fumigaciones-02] convex/fumigaciones.ts:758 — `_migrationBackfillOrganizacionId` sin auth gate. Fix: añadir `await requireSuperAdmin(ctx)` + env check.
- [maintenance-01] convex/maintenance.ts:715 — `_migrationBackfillOrganizacionId` sin auth gate. Fix: añadir `await requireSuperAdmin(ctx)` + env check.
- [route_events-01] convex/route_events.ts:145 — `_migrationBackfillOrganizacionId` sin auth gate. Fix: añadir `await requireSuperAdmin(ctx)` + env check.
- [rutas-02] convex/rutas.ts:266 — `_migrationBackfillOrganizacionId` sin auth gate. Fix: añadir `await requireSuperAdmin(ctx)` + env check.

## P1
- [cleaning-04] convex/cleaning.ts:171-194 — `addAssignment` también falta `await requireProjectAccess` pero solo via sala.proyecto_id (línea 185 sí lo llama). Si requireWriteRole se añade, ok. Pero sin requireWriteRole un enterprise puede crear assignments — combo con P0 cleaning-01.
- [empleados-02] convex/empleados.ts:131-140 — Race condition en validación de unicidad de cédula. Lookup por `by_cedula` luego insert; dos calls concurrentes con misma cédula pueden insertarse. OCC reduce ventana pero no hay constraint a nivel schema. Fix: validar tras insert con try/catch y manual revert, o aceptar OCC retry.
- [inventario-01] convex/inventario.ts:29-36 — `generateCodigo` retorna `MAT-{count+1}` basado en `items.length+1`. Race condition: dos calls concurrentes generan mismo código. No hay unicidad enforced. Fix: server-side generar codigo dentro de `add` mutation, no en query.
- [route_progress-02] convex/route_progress.ts:189-198 — `start` cierra route_progress `en_progreso` con mismo vehiculo_id pero NO emite evento `route_events` `ruta_terminada_anticipadamente` para auditoría. Cierre silencioso de rutas activas. Fix: insertar route_event al patch.
- [route_progress-03] convex/route_progress.ts:7-22 — `cleanupStaleInProgress` cierra route_progress pero NO libera estado del vehículo asociado (queda en `en_ruta`). Inconsistencia post-cleanup. Fix: tras cerrar cada RP, evaluar si hay otro RP en_progreso del mismo vehículo; si no, patchear vehiculo.estado="disponible".
- [route_events-02] convex/route_events.ts:96-125 — `attachPhotoToParada` retorna `null` si no encuentra el evento parada_completada (línea 116) en vez de emitir nuevo evento como fallback que dice el comentario (línea 95). Fotos del queue offline se pierden silenciosamente. Fix: emitir nuevo evento `parada_completada` con la foto si target===null.
- [maintenance-02] convex/maintenance.ts:441-442 — `listReports` para enterprise/conductor no filtra por `organizacion_id`; retorna `all.filter((r) => !r.proyecto_id || r.proyecto_id === scope.proyectoId)` — un conductor de org A ve reports de org B si comparten proyecto_id (improbable pero posible cross-tenant). Fix: filtrar por `scope.organizacionId` además de proyecto.
- [route_reports-01] convex/route_reports.ts:91-106 — `countAll` es `internalQuery`, gated por `assertPurgeAllowed()` pero no por auth. internalQuery solo accesible vía CLI; aceptable defense-in-depth pero debería loggear `who` cuando ALLOW_PURGE=1.
- [organizaciones-01] convex/organizaciones.ts:521-617 — `recomputeStorage` paginado: si org tiene >5000 fotos por tabla, retorna `partial=true` pero PATCHEA `storage_bytes_used=total` con valor incompleto. Sobrescribe contador correcto con menor valor → cobro de overflow incorrecto. Fix: si `partial`, NO patchear storage_bytes_used, solo retornar el conteo parcial y loggear warn.
- [schema-01] convex/schema.ts:372-390 — `inventario_movimientos` carece de índice `by_organizacion`. Queries en `inventario.getConsumoPorTipo` y `getHistorialComprasPorMes` filtran in-memory por org → full scan + filter. Performance hit notable en orgs grandes. Fix: añadir `.index("by_organizacion", ["organizacion_id"])` y `.index("by_org_tipo_fecha", ["organizacion_id", "tipo_movimiento", "fecha"])`.
- [inventario-02] convex/inventario.ts:565-580 — `registrarCompra` hace 3 patches secuenciales (precio_unitario, proveedor, cantidad_disponible) en el mismo doc. Convex serializa OK, pero hay tres write conflicts potenciales en concurrencia. Fix: combinar en un solo `ctx.db.patch(item_id, { precio_unitario, proveedor, cantidad_disponible })`.
- [fumigaciones-03] convex/fumigaciones.ts:687-755 — `createReport` no llama `requireModulo(ctx, "FUM")` (sí llama `requireWriteRole` línea 708). Inconsistente con resto. Fix: añadir `await requireModulo(ctx, "FUM");`.
- [maintenance-03] convex/maintenance.ts:594-641 — `createReport` no llama `requireModulo(ctx, "MTO")`. Fix: añadir `await requireModulo(ctx, "MTO");`.
- [inventario-03] convex/inventario.ts:375-465 — `asignarDesdeAlmacen` no llama `requireModulo(ctx, "INV")` (solo `requireWriteRole`). Fix: añadir `await requireModulo(ctx, "INV");`.
- [inventario-04] convex/inventario.ts:544-598 — `registrarCompra` no llama `requireModulo(ctx, "INV")`. Fix: añadir.
- [inventario-05] convex/inventario.ts:755-818 — `registrarConsumo` no llama `requireModulo(ctx, "INV")`. Fix: añadir.
- [fumigaciones-04] convex/fumigaciones.ts:534-560 — `deletePhoto` no llama `requireModulo(ctx, "FUM")`. Fix: añadir.

## P2
- [schema-02] convex/schema.ts:278 — `route_events.timestamp` es `v.string()` pero CLAUDE.md indica "Always use number type (ms since epoch), never string". Inconsistencia con resto del schema (vehiculos.gps_ultima_actualizacion etc). Fix: migrar a `v.number()`.
- [schema-03] convex/schema.ts:177-200 — `asignaciones_rutas.estado` es `v.string()` libre — sin union literal. State machine en `asignaciones.updateEstado` valida pero `add`/`update` aceptan cualquier string. Fix: cambiar a `v.union(v.literal("asignada"), v.literal("programada"), v.literal("en_progreso"), v.literal("completada"), v.literal("cancelada"))`.
- [schema-04] convex/schema.ts:147-174 — `rutas.estado` es `v.string()` sin union. Mismo problema. Fix: usar union literal.
- [schema-05] convex/schema.ts:104-144 — `vehiculos.estado` y `tipo_servicio` son `v.string()` sin union literal. Fix: usar union literal con valores documentados.
- [schema-06] convex/schema.ts:344-359 — `inventario.tipo_articulo` es `v.string()` sin union (debería ser "herramienta"|"insumo"|"equipo"|"uniforme"). Fix: usar union literal.
- [schema-07] convex/schema.ts:463-482 — `maintenance_tasks.estado/tipo/prioridad` son `v.string()` sin union. Fix: usar union literal.
- [inventario-06] convex/inventario.ts:98-177 — `add` mutation acepta `codigo: v.string()` desde el cliente sin validar unicidad ni formato. Cliente puede pasar codigo arbitrario duplicado. Fix: generar codigo server-side, ignorar el del cliente; validar `by_codigo` único per org (añadir index).
- [cleaning-05] convex/cleaning.ts:233-245 — `deleteAssignment` borra fotos con `await ctx.storage.delete` dentro de loop sin try/catch (línea 233 sí lo tiene). Si `incrementOrgStorage` falla, drift en storage counter. Fix: wrap todo el loop en try/catch granular por foto.
- [route_progress-04] convex/route_progress.ts:163-183 — `start` valida vehiculo y asignación pero no valida que el RUTA del arg corresponda a la asignación. Conductor podría empezar progress con `ruta_id` distinto a `asignacion.ruta_id`. Fix: validar `asignacionCheck.ruta_id === args.ruta_id`.
- [empleados-03] convex/empleados.ts:30-40 — `stripPII` expone `salario` solo a super_admin pero NO a admin propio. Admin de la org puede no ver salarios de sus propios empleados. Verificar si es by-design; si no, expandir a admin.
- [organizaciones-02] convex/organizaciones.ts:181-203 — `countOrgUsage` filtra usuarios con `u.activo !== false`. Esto incluye `u.activo === undefined` como activos (legacy users). Schema obliga `activo: v.boolean()` pero datos legacy pueden carecer. Fix: explícito `u.activo === true`.
- [route_events-03] convex/route_events.ts:67-79 — Idempotency check usa `q.eq("parada_index", args.parada_index)` sin verificar que tipo_evento y asignacion_id matchen exactamente. Si parada_index existe en otro evento del mismo asignacion_id pero con tipo distinto, devuelve mal ID. Filter aclara, ok. No issue real.
- [fumigaciones-05] convex/fumigaciones.ts:534-560 — `deletePhoto` borra storage ANTES de decrementar counter ANTES de delete row. Si una de las 3 ops falla, drift. Fix: orden: delete row → delete storage → incrementOrgStorage; o usar transaction-style con rollback. Convex no soporta xact compuestos así que orden defensivo.
- [maintenance-04] convex/maintenance.ts:441-444 — `listReports` para admin retorna `all` sin filtrar por org. Admin de org A ve reports de org B porque la query no scopes por org. Fix: filtrar `all` por `scope.organizacionId` cuando admin.
